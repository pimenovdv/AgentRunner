# Пошаговый план разработки Runner-сервиса (Проигрыватель / Player) на FastAPI и LangGraph

На основе системной аналитики и архитектуры мультиагентной платформы, Runner-сервис представляет собой изолированный движок микро-исполнения агентов. Его главная задача — получить декларативный JSON-манифест агента, инициализировать контекст, динамически построить и выполнить внутренний граф (на базе LangGraph) и гарантированно вернуть результат, соответствующий строгим JSON-контрактам.

Ниже представлен подробный step-by-step TODO лист для разработки данного сервиса.

---

- [x] **1.1. Создание репозитория и настройка окружения:**
  - Инициализировать проект с использованием `Poetry` или `uv`.

  - Установить утилитарные библиотеки: `httpx` (для REST API и MCP), `tenacity` (для retries), `jsonpath-ng` (для парсинга REST ответов), `confluent-kafka` (для интеграции с Kafka).

## Этап 2: Проектирование API контрактов и моделей данных (Pydantic)
- [x] **2.1. Разработка моделей для API эндпоинта:**
  - Создать Pydantic-модель `ExecuteRequest` для `POST /api/v1/player/execute`, включающую поля `execution_id`, `agent_manifest`, `input_context`, `execution_limits` (`max_tokens`, `timeout_ms`).
  - Создать Pydantic-модель `ExecuteResponse`, описывающую структуру успешного ответа (`status`, `output_data`, `telemetry`).
- [x] **2.2. Моделирование структуры JSON-манифеста агента (`agent_manifest`):**
  - Описать модели секций манифеста: `input_schema`, `output_schema` (с поддержкой JSON Schema).
  - Создать модели для секции `prompts` (System Instructions, Guardrails).
  - Реализовать модели для структуры графа (Micro-Graph): `nodes` (типы узлов: `reasoning`, `tool_execution`, `data_transformation`), `edges`, `conditions`.
  - Создать базовые и полиморфные модели для инструментов (Tools):
    - `BuiltinTool` (утилиты).
    - `RestApiTool` (с полями `method`, `base_url`, `parameters_schema`, маппингом заголовков).
    - `McpTool` (настройки подключения к серверу MCP).
    - `KafkaTool` (topic, role, `payload_schema`).
- [ ] **2.3. Дополнительные модели (Missing Models):**
  - Создать Pydantic-модели для `LlmConfig` (настройки LLM: provider, model_name, temperature).
  - Создать модель для LangGraph State (`State`, `Message`, `ToolCall`).
  - Создать модель для настройки Retry-политик (Exponential Backoff, Max Retries).

## Этап 3: Подсистема интеграции с инструментами (Tools Subsystem)
- [ ] **3.1. Реализация подсистемы Built-in инструментов:**
  - Разработать локальные безопасные утилиты: калькулятор, форматировщик дат, генератор криптографических хешей, парсер регулярных выражений.
  - Обернуть их в формат инструментов LangChain/LangGraph.
- [ ] **3.2. Механизм декларативного REST API вызова:**
  - Создать HTTP-клиент (на базе `httpx.AsyncClient`).
  - Реализовать валидацию аргументов, сгенерированных LLM, через `parameters_schema` инструмента.
  - Реализовать интерполяцию шаблонов (вставка параметров в URL или тело запроса).
  - Разработать механизм получения секретов (из Vault или мок-хранилища) для подстановки в заголовки авторизации.
  - Написать логику извлечения данных из большого ответа REST API с использованием `JSONPath` для экономии контекста (токенов).
- [ ] **3.3. Интеграция Model Context Protocol (MCP) клиента:**
  - Реализовать базовый MCP-клиент, поддерживающий транспортные протоколы (HTTP+SSE или stdio).
  - Реализовать вызов `tools/list` на этапе инициализации агента для динамического проброса манифеста инструментов в LLM.
  - Реализовать обработчик `tools/call` для передачи запросов от LLM к MCP-серверу и возврата результата в контекст.
- [ ] **3.4. Интеграция с Apache Kafka:**
  - Настроить Kafka Producer: валидация сформированного LLM сообщения через `payload_schema` (JSON Schema) и асинхронная отправка в указанный топик.
  - *(Опционально для Runner'а)* Настроить интеграцию с Confluent Schema Registry для строгой сериализации.

## Этап 4: Интеграция LangGraph и конструирование Micro-Graph
- [ ] **4.1. Разработка State Machine:**
  - Определить класс состояния (State TypedDict или Pydantic-модель) для узлов графа LangGraph (локальное состояние агента, хранение контекста `input_context`, история сообщений LLM, результаты вызова инструментов).
- [ ] **4.2. Механизм динамической генерации графа:**
  - Написать фабрику (builder), которая принимает JSON-манифест агента и конвертирует его в `StateGraph` из библиотеки LangGraph.
  - Реализовать трансляцию секции `nodes` манифеста в реальные python-функции (узлы графа LangGraph).
  - Настроить маршрутизацию (Conditional Edges): транслировать секцию `edges` и `conditions` манифеста в функции маршрутизации (Skip Logic), оценивающие состояние и принимающие решение о следующем узле.
- [ ] **4.3. Реализация базовых типов узлов (Nodes logic):**
  - **Reasoning Node:** Вызов LLM (Chat Model) с компиляцией системных промптов (System Instructions), истории и внедрением доступных на данном этапе инструментов (Tools binding).
  - **Tool Execution Node:** Парсинг запроса LLM (Tool Call), диспетчеризация вызова конкретного инструмента (из Этапа 3) и добавление ответа (Tool Message) в историю.

## Этап 5: Паттерны отказоустойчивости (Resilience & Fallbacks)
- [ ] **5.1. Circuit Breakers и Retries:**
  - Обернуть вызовы к LLM-провайдерам, REST API и MCP инструментам в декораторы `tenacity` с экспоненциальной задержкой (Exponential Backoff) для обработки сетевых сбоев (429, 50x ошибки).
- [ ] **5.2. Ограничение итераций (Loop Bounds):**
  - Реализовать защиту от бесконечных циклов в LangGraph, задав `recursion_limit` при компиляции графа или внедрив счетчик шагов в состояние агента (State).
  - При превышении лимита принудительно направлять выполнение на узел ошибки (Fallback Node).
- [ ] **5.3. Механизм авто-коррекции (Self-Correction & Reflection):**
  - Разработать логику валидации промежуточных ответов LLM (особенно перед вызовом инструментов или на выходе).
  - Если сгенерированный JSON не проходит валидацию Pydantic, перехватить ошибку `ValidationError`.
  - Сформировать системное сообщение с текстом ошибки Pydantic и вернуть его в LLM с просьбой исправить структуру ответа, создав цикл рефлексии (Reflection loop).
